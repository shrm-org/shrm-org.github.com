Current State of Content Management at SHRM

  • Mostly Sharepoint.
  • Also Drupal.
  • And WordPress.
  • And custom frameworks.
  • And static sites.

Current State: Technical Challenges

  • Rigidly presentational markup. (Only one rendering of each document is possible.)

  • Content production requires extraordinary manual effort. (Content sharing and reuse is driven by cut-and-paste. RSS feeds and email newsletters are manually created. Document workflow and revision tracking happens manually.)

  • Web design and production requires extraordinary manual effort. (Most documents require manual cleanup on import. Design standards can only be enforced socially. Design changes must be independently and manually applied to each document.)

Current State: Technical Challenges (cont.)

  • Limited meta-data. (Can't tell which documents link to this one. Can't tell whether or where a given image or media is used. Can't embargo or "de-publish" a document. Can't identify when a document was last updated.)

  • Arbitrary and obtrusive technical constraints. (Arbitrary limit on the number of documents per section. Visual design is constrained by template architecture. Control over the structure and content of generated HTML is limited. Cannot easily import or export documents and templates between environments.)

  • Incomplete. (Images, audio, video, PDF and other media assets are managed outside the system. Many micro-sites and sub-sites are managed outside the system.)

Current State: Social Challenges

  • Diverse contributors shoehorned into a one-size-fits-all interface. (Product landing/conversion pages, breaking news, magazine content, research reports, toolkits, Q&A, help pages, etc. all share the same editorial interface.)

  • Power users find the editorial interface awkward. They avoid it whenever they can. (The entire editorial pipeline exists outside the CMS. Content is only imported into the CMS for publication.)

  • Casual users find the editorial interface difficult to understand. They rely on web-production staff to import and maintain content.

Current State: Social Challenges (cont.)

  • The most recent CMS replacement was painful for content contributors. (It required signficant and unplanned support from the content creators. The migration degraded the quality of their content--introducing many errors and even more broken links. The deployment created an enormous disruption to their work, and stretched out over months due to purely technical issues.) The clients' initial POV was skepticm and fear.

Sherman's Architectural Foundation

Sherman's Architectural Foundation

Sherman uses a RESTful Service Oriented Architecture (SOA).

(Don't panic. We'll explain what that means.)

Service Oriented Architecture?

  • A federation of subsystems, each with a specific role and set of responsibilities.

  • Each subsystem is atomic, providing a complete "service" in and of itself.

  • Each subsystem is independent, capable of being used in isolation (or with non-Sherman components).

  • Each subsystem is integrated, designed to work in concert to deliver a comprehensive content management solution.

RESTful?

  • REST = Representational State Transfer, a pattern for the design of web service APIs based on the architecture of the World Wide Web itself.

  • Question: What has allowed the WWW to become such a massively scalable, globally distributed, robust, diverse, resilant and flexible system?

  • Answer: The use of a (1) uniform, (2) stateless, (3) cachable, (4) layered, (5) clientâ€“server architecture.

REST: Computers should use the web the way people use the web.

  • People browse a web site by (1) starting at the homepage, (2) discovering links that point to other pages, (3) following those links to view content, and (4) occassionally using web forms to create new or modify existing pages.

  • RESTful clients use a web service by (1) starting at a known "entry point", (2) discovering URLs that point to other resources , (3) following those URLs to fetch data and (4) occassionally using HTTP methods to create new or modify existing resources.

Why REST?

  1. The REST architetural style confers the same technical advantages to web services that it did to the web itself.

  2. RESTful web services can leverage the existing web ecosystem (clients, servers, tools, monitors, hosts, etc.).

Sherman's Primary Services

Sherman decomposes content management into three major functions:

  1. Content Display Service - user-facing; responsible for the final rendering and delivery of content

  2. Content Management Service - contributor-facing; responsible for helping capture, create, organize, maintain and publish content

  3. Content Repository Service - intermediary between content management and content display; responsible for housing and distributing published content

Sherman's Primary Services

Sherman's Primary Services

These services interoperate across RESTful web service APIs.

Different Interfaces for Different Needs

To deliver content from diverse contributors to diverse consumers, Sherman provides multiple content management and content display interfaces.

Each CMA (content management application) and CDA (content display application) is backed by a centralized, core management or display service.

Technical Components

Sherman's structure isn't that different from Sharepoint's.

Sharepoint has the same functional components as the Sherman CMS, but they bundled together into a monolithic system.

...but it is more flexible.

By partioning the CMS into independent services that interoperate across standard APIs, there are a lot more choices to be made about Sherman's implementation.

Sherman's Technology Stack: CMA

  • Rails application running on the Ruby platform.

  • Backed by a Neo4J datastore..

Rationale?

  • Ruby-on-Rails is a popular and powerful tool for rapidly developing web forms.
  • Neo4J is optimized to represent dynamic and highly-interconnected data while retaining fully transactional, enterprise-strength ACID reliabilty and data integrity.

Sherman's Technology Stack: CR

  • Riak, a document-oriented key-value store.
  • Documents stored in a semantic structure based on Dublin Core meta-data and conventional "microformats"

Rationale?

  • Riak is a high-performance, high-availablity, cloud-ready, easily-managed. distributed database.
  • Microformats standardize common meta-data structures for broad interoperabilty.

Sherman's Technology Stack: CDA

  • Dust.js templating enging running on Node.js/CoffeeScript platform.
  • Exposing content via HTML5, JSON and other interfaces.

Rationale?

  • Consistent use of JavaScript allows display engine to run on server, in a web browser or on a mobile device.
  • Also provides web designers and "front-end" engineers with more autonomy. The skills developed using client-side JavaScript and ActionScript (Flash) can now be leverged for server-side and mobile development.

Build vs. Buy?

This starts to look like a build-versus-buy scenario.

Is this a custom CMS? Is that more work? Is that more risk? Wouldn't it be better to get an off-the-shelf CMS?

Build vs. Buy?

SHRM, like any organization that produces content at a large scale, has sophsticated and specialized needs.

Even an off-the-shelf CMS such as Sharepoint requires substantial customization to fit into our environment.

Build vs. Buy?

A closer analysis suggests that Sherman doesn't require much more customization than our existing implementation.

Implementation Plan

Sherman modular architure allows flexible deployment/roll-out:

  • As a complete and fully independent application.

  • By connecting Sherman's Content Management inteface to Sharepoint's Cntent Repository.

  • By connecting Sherman's Content Display inteface to Sharepoint's Cntent Repository.

  • Arbitrary combinations of the above.

Implementation Plan (Cont.)

Leverages Sherman's modularity to perform a phased/incremental delivery.

Currently working on two major milestones:

  1. HR News CMA. [August]

  2. Mobile SHRM Online (Homepage) [October]
